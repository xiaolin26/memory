# 协作开发指南

> **作用**：指导团队如何使用 Memory 系统协作开发

---

## 📁 Memory 系统文件

| 文件 | 作用 | 更新频率 | 是否提交Git |
|-----|------|---------|-----------|
| **MEMORY.md** | 项目历史、技术决策 | 每次重大变更 | ✅ 是 |
| **CONSTITUTION.md** | 开发规范、约定 | 调整规范时 | ✅ 是 |
| **key.md** | 敏感信息记录 | 新增敏感信息时 | ❌ 否 |

---

## 🔄 开发流程

### 场景1：新成员加入

1. **阅读文档**（30分钟）
   - docs/README.md
   - MEMORY.md
   - CONSTITUTION.md
   - docs/环境配置指南.md

2. **配置环境**
   ```bash
   # 获取配置信息（通过安全渠道）
   cp [配置模板] [实际配置]
   # 填写实际配置
   ```

3. **启动项目**
   ```bash
   [启动命令]
   ```

---

### 场景2：开发新功能

**开发前**（5分钟）：
- [ ] 阅读 MEMORY.md
- [ ] 确认需求

**开发中**：
- [ ] 遵循代码规范
- [ ] 敏感信息 → key.md
- [ ] 编写测试

**开发后**（10分钟）：
- [ ] 更新 MEMORY.md
- [ ] 更新功能文档
- [ ] 更新版本号
- [ ] 运行测试

**提交示例**：
```bash
git add .
git commit -m "feat([模块]): 添加XX功能

- 实现XX功能
- 编写测试

Co-Authored-By: [姓名] <[邮箱]>"
```

---

### 场景3：修复 Bug

1. **定位问题**
2. **修复并测试**
3. **更新 MEMORY.md**（记录Bug和修复）
4. **更新版本号**（PATCH）

---

### 场景4：重构代码

1. **评估风险和收益**
2. **讨论重构方案**
3. **保持功能不变**
4. **更新 MEMORY.md**（记录原因和方案）
5. **更新版本号**（MINOR）

---

## 📋 文档更新规范

## 📊 快速参考

### 何时更新文档

| 操作 | 更新文件 |
|-----|---------|
| 代码变更 | MEMORY.md |
| 功能变更 | 功能文档 |
| 规范变更 | CONSTITUTION.md |
| 新增敏感信息 | key.md（本地） |

### 核心原则

- 📝 **及时更新**：变更后立即更新
- 🎯 **保持简洁**：记录重要信息
- 🤝 **团队协作**：确保团队遵循规范
- 🔄 **持续改进**：根据反馈优化

---

## 🆘 常见问题

**Q: 必须记住所有内容吗？**

A: 不用。记住：
- 开发前：读 MEMORY.md + CONSTITUTION.md
- 开发后：更新 MEMORY.md + 功能文档
- 敏感信息：记录到 key.md（不提交）

**Q: 遗忘更新怎么办？**

A: 立即补充更新，提交时说明是补充更新。

**Q: 不知道更新哪个文档？**

A: 代码变更 → MEMORY.md，功能变更 → 功能文档
